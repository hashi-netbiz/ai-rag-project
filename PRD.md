# RAG-Based Role-Based Access Control Chatbot — PRD

**Version:** 1.2 | **Date:** 2026-03-25 | **Status:** MVP Complete

---

## 1. Executive Summary

This project delivers a production-quality AI chatbot that enforces role-based access control (RBAC) over company data through a Retrieval-Augmented Generation (RAG) pipeline. Employees interact with a conversational interface and receive answers grounded exclusively in documents their role is authorized to access — with every response citing its source.

The system authenticates users via JWT, resolves their role, and restricts Pinecone vector search to permitted document namespaces at query time. Retrieval is improved by a Pinecone native reranker (`bge-reranker-v2-m3`) that narrows k=10 candidates down to the top 3 most relevant chunks before generation. Responses are generated by Llama 3.3 70B (via Groq) and evaluated via Ragas and Langsmith.

**MVP Status:** Complete. All 6 roles are live, RBAC is enforced, reranking is active, and the professional Next.js 15 frontend is running on port 3001.

## 2. Mission

**Mission Statement:** Democratize access to company knowledge while strictly enforcing information boundaries — every employee gets the right answer from the right data, nothing more.

**Core Principles:**

1. **Access by role, not by trust** — RBAC is enforced at the retrieval layer, not just the UI
2. **Grounded responses only** — the LLM never answers beyond the retrieved context
3. **Transparency** — every answer cites the source document and section
4. **Observability** — every query is traced and every answer is measurable
5. **Simplicity** — no over-engineering; clean, maintainable code using standard tooling

## 5. User Stories

1. **As a Finance Analyst**, I want to ask "What was our Q3 gross margin?" and receive an accurate answer with a citation to the quarterly report, so that I can prepare presentations without manually searching documents.

2. **As a General Employee**, I want to ask "How many sick days am I entitled to?" and get a clear answer from the employee handbook, so that I don't need to contact HR for basic policy questions.

3. **As a C-Level Executive**, I want to ask cross-department questions and get a synthesized answer, so that I have full business visibility in one interface.

4. **As an HR Business Partner**, I want to query employee attendance and performance data, so that I can quickly prepare for review meetings.

5. **As a Marketing Manager**, I want to ask about campaign ROI and customer acquisition costs, so that I can brief my team without digging through quarterly reports.

6. **As a General Employee**, I want the chatbot to say "I don't have access to that information" if I ask about financials, so that I trust the system is properly securing sensitive data.

7. **As an Engineering Lead**, I want to ask about our deployment process or tech stack, so that I can onboard new engineers faster.

8. **As any user**, I want to see which document each part of the answer came from, so that I can verify the information and read the full source if needed.

## 6. Core Architecture & Patterns

### High-Level Architecture

```text
User → Next.js 15 UI (port 3001)
         ├── /api/auth/login    → FastAPI /auth/login
         ├── /api/auth/me       → FastAPI /auth/me
         └── /api/chat/query    → FastAPI /chat/query
                                          │
                               FastAPI Backend (port 8000)
                                          ├── Auth (JWT)
                                          ├── RBAC (role → allowed departments)
                                          └── RAG Service
                                                ├── Google Embeddings (query)
                                                ├── Pinecone (k=10 filtered retrieval)
                                                ├── Pinecone Reranker → top 3
                                                ├── Groq LLM (Llama 3.3 70B)
                                                └── Langsmith (tracing)
```

**Proxy pattern:** The browser calls `/api/*` on the same Next.js origin — no CORS headers required. Next.js API routes forward requests server-side to `http://localhost:8000`.

### Directory Structure

```text

├── backend/

│   └── app/
│       ├── main.py              ← FastAPI app + CORS
│       ├── config.py            ← pydantic settings
│       ├── auth/
│       ├── chat/
│       │   └── rag_service.py   ← RAG pipeline + reranking
│       ├── rbac/
│       └── vector_store/

    ├── stores/                  ← Zustand auth + chat stores
    └── components/
```

### Key Design Patterns

- **RBAC enforced at retrieval** — Pinecone metadata filter `{"department": {"$in": allowed}}` applied before any data is returned; the LLM never sees unauthorized content
- **Reranking after retrieval** — `bge-reranker-v2-m3` cross-encoder reorders k=10 candidates; only top 3 reach the LLM, improving context precision
- **Stateless auth** — JWT carries role claim; server validates on every request
- **Dependency injection** — FastAPI `Depends(get_current_user)` for auth on protected routes
- **Next.js proxy** — browser never touches port 8000; API routes forward server-side

## 7. Features

### Feature 1: JWT Authentication

- `POST /auth/login` accepts `{username, password}`, returns `{access_token, token_type, role}`
- `GET /auth/me` returns current user info
- Token contains: `sub` (username), `role`, `exp`
- 6 demo users pre-seeded at app startup (one per role)

### Feature 2: Role-Based RAG Query with Reranking

- `POST /chat/query` (authenticated)
- Flow: extract role from JWT → get allowed departments → embed query → Pinecone filtered search (k=10) → Pinecone rerank (`bge-reranker-v2-m3`, top 3) → Groq Llama 3.3 70B generation → return answer + sources
- Reranking fallback: if Pinecone inference API fails, returns first 3 docs unchanged

### Feature 3: Source Citations

- Every response includes `sources: [{file: str, section: str}]` — maximum 3 entries (post-rerank)
- Frontend renders citations as collapsible chips below each answer

### Feature 4: Data Ingestion Pipeline

- One-time script: load → chunk → embed → upsert to Pinecone
- Markdown: `MarkdownHeaderTextSplitter` on `##`/`###` boundaries, then `RecursiveCharacterTextSplitter(chunk_size=800, chunk_overlap=100)`
- CSV (`hr_data.csv`): each row = one document
- All chunks tagged with `{department, source_file, doc_type, chunk_id}`
- Embedding model: `gemini-embedding-2-preview` (768-dim)

### Feature 5: Observability

- Langsmith: automatic tracing of every RAG chain invocation via `@traceable`
- Ragas: offline evaluation script scoring faithfulness, answer relevancy, context precision, context recall
- Results exported as `backend/evaluation_results.csv`

### Feature 6: Professional Next.js 15 Chat UI

- Login screen with animated card (Motion 12), shadcn/ui components
- Sidebar with role badge, dark mode toggle, logout
- Chat with typing indicator, empty state with suggestion chips, collapsible source citations
- Markdown rendering via `react-markdown` + `remark-gfm`
- Dark mode with localStorage persistence; TypeScript strict mode, zero type errors

### Feature 7: End-to-End Regression Suite

- `test_e2e.sh` — 42 automated tests using `curl`
- Covers: service health, all 6 auth users, RBAC denied, RBAC granted, source citations, Next.js proxy routes, TypeScript check
- Exit 0 = all pass; Exit 1 = any failure

## 9. Security & Configuration

### Authentication

- JWT with `HS256` algorithm, configurable expiry (default 60 min)
- Passwords stored as bcrypt hashes (in-memory for MVP)
- `get_current_user` FastAPI dependency enforces auth on all `/chat/*` routes

### RBAC

- Role resolved from JWT claim — never from client-supplied headers
- Pinecone metadata filter applied server-side before retrieval
- LLM prompt contains only authorized, reranked context (top 3 chunks)

### Configuration (Environment Variables)

```env
GROQ_API_KEY=
GOOGLE_API_KEY=
PINECONE_API_KEY=
PINECONE_INDEX_NAME=rag-rbac-chatbot
JWT_SECRET_KEY=
JWT_ALGORITHM=HS256
JWT_EXPIRE_MINUTES=60
LANGCHAIN_API_KEY=
LANGCHAIN_TRACING_V2=true
LANGCHAIN_PROJECT=rag-rbac-chatbot
```

### Security Scope

- Server-side role enforcement, secrets in `.env` (gitignored), CORS restricted to `http://localhost:3001`
- bcrypt password hashing, Next.js proxy (browser never directly calls FastAPI)
- HTTPS/TLS, rate limiting, and audit logging are deferred to production infrastructure

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- |
| Pinecone metadata filter bypass | Low | High | Enforce filter server-side only; never trust client-supplied filters |
| LLM hallucination beyond retrieved context | Low | Medium | Prompt instructs model to answer from context only; reranking reduces noise |
| Google embedding API rate limits | Low | Medium | Batch embedding calls; add retry with exponential backoff |
| Groq API downtime | Low | High | User-friendly fallback message; consider model fallback |
| JWT secret exposure | Low | Critical | Rotate secret immediately; use strong random value in production |
| Pinecone reranker API failure | Low | Low | Fallback returns `docs[:3]` — pipeline continues uninterrupted |
