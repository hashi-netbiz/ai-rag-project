# Product Requirements Document
# RAG-Based Role-Based Access Control Chatbot

**Version:** 1.0
**Date:** 2026-03-24
**Status:** Approved for Development

---

## 1. Executive Summary

This project delivers a production-quality AI chatbot that enforces role-based access control (RBAC) over company data through a Retrieval-Augmented Generation (RAG) pipeline. Employees interact with a conversational interface and receive answers grounded exclusively in documents their role is authorized to access — with every response citing its source.

The system authenticates users via JWT, resolves their role, and restricts Pinecone vector search to permitted document namespaces at query time. Responses are generated by Llama 3.1 70B (via Groq) and evaluated continuously via Ragas and Langsmith.

**MVP Goal:** Deploy a working chatbot serving 6 distinct company roles, each with enforced data boundaries, source citations, and full observability — built on Python/FastAPI (uv), React, Pinecone, and Groq.

---

## 2. Mission

**Mission Statement:** Democratize access to company knowledge while strictly enforcing information boundaries — every employee gets the right answer from the right data, nothing more.

**Core Principles:**
1. **Access by role, not by trust** — RBAC is enforced at the retrieval layer, not just the UI
2. **Grounded responses only** — the LLM never answers beyond the retrieved context
3. **Transparency** — every answer cites the source document and section
4. **Observability** — every query is traced and every answer is measurable
5. **Simplicity** — no over-engineering; clean, maintainable code using standard tooling

## 5. User Stories

1. **As a Finance Analyst**, I want to ask "What was our Q3 gross margin?" and receive an accurate answer with a citation to the quarterly report, so that I can prepare presentations without manually searching documents.

2. **As a General Employee**, I want to ask "How many sick days am I entitled to?" and get a clear answer from the employee handbook, so that I don't need to contact HR for basic policy questions.

3. **As a C-Level Executive**, I want to ask cross-department questions like "How did marketing spend compare to revenue growth last quarter?" and get a synthesized answer, so that I have full business visibility in one interface.

4. **As an HR Business Partner**, I want to query employee attendance and performance data, so that I can quickly prepare for review meetings.

5. **As a Marketing Manager**, I want to ask about campaign ROI and customer acquisition costs, so that I can brief my team without digging through quarterly reports.

6. **As a General Employee**, I want the chatbot to tell me "I don't have access to that information" if I ask about financials, so that I trust the system is properly securing sensitive data.

7. **As an Engineering Lead**, I want to ask about our deployment process or tech stack, so that I can onboard new engineers faster.

8. **As any user**, I want to see which document each part of the answer came from, so that I can verify the information and read the full source if needed.

### Key Design Patterns
- **RBAC enforced at retrieval** — Pinecone metadata filter `{"department": {"$in": allowed}}` applied before any data is returned; the LLM never sees unauthorized content
- **Stateless auth** — JWT carries role claim; server validates on every request
- **Dependency injection** — FastAPI `Depends(get_current_user)` for auth on protected routes
- **Chain of responsibility** — LangChain `RetrievalQAWithSourcesChain` separates retrieval from generation

## 7. Features

### Feature 1: JWT Authentication
- `POST /auth/login` accepts `{username, password}`, returns `{access_token, token_type, role}`
- `GET /auth/me` returns current user info
- Token contains: `sub` (username), `role`, `exp`
- 6 demo users pre-seeded at app startup (one per role)

### Feature 2: Role-Based RAG Query
- `POST /chat/query` (authenticated)
- Flow: extract role from JWT → get allowed departments → embed query (Google) → Pinecone filtered search → LLM generation (Groq) → return answer + sources
- Pinecone filter: `{"department": {"$in": ["finance", "general"]}}` (example for finance role)

### Feature 3: Source Citations
- Every response includes `sources: [{file: str, section: str}]`
- Frontend renders citations as collapsible chips below each answer

### Feature 4: Data Ingestion Pipeline
- One-time script: load → chunk → embed → upsert to Pinecone
- Markdown files: `MarkdownHeaderTextSplitter` on `##`/`###` boundaries, then `RecursiveCharacterTextSplitter(chunk_size=800, chunk_overlap=100)`
- CSV (hr_data.csv): each row = one document
- All chunks tagged with `{department, source_file, doc_type, chunk_id}`

### Feature 5: Observability
- Langsmith: automatic tracing of every RAG chain invocation
- Ragas: offline evaluation script scoring faithfulness, answer relevancy, context precision, context recall
- Results exported as CSV

### Feature 6: Chat UI
- Login screen with username/password form
- Chat interface: message history + input box
- Role badge shown in header (user knows what role they're logged in as)
- Source citation chips below each AI response

## 9. Security & Configuration

### Authentication
- JWT with `HS256` algorithm, configurable expiry (default 60 min)
- Passwords stored as bcrypt hashes (in-memory for MVP)
- `get_current_user` FastAPI dependency enforces auth on all `/chat/*` routes

### RBAC
- Role resolved from JWT claim — never from client-supplied headers
- Pinecone metadata filter applied server-side before retrieval
- LLM prompt contains only authorized context

### Configuration (Environment Variables)
```env
GROQ_API_KEY=                  # Groq console
GOOGLE_API_KEY=                # Google AI Studio
PINECONE_API_KEY=              # Pinecone dashboard
PINECONE_INDEX_NAME=rag-rbac-chatbot
JWT_SECRET_KEY=                # 32-char random string
JWT_ALGORITHM=HS256
JWT_EXPIRE_MINUTES=60
LANGCHAIN_API_KEY=             # Langsmith
LANGCHAIN_TRACING_V2=true
LANGCHAIN_PROJECT=rag-rbac-chatbot
```

### Security Scope
- ✅ Server-side role enforcement
- ✅ Secrets in `.env` (gitignored)
- ✅ CORS restricted to `http://localhost:3000`
- ✅ bcrypt password hashing
- ❌ HTTPS/TLS (deferred — handled by reverse proxy in production)
- ❌ Rate limiting (deferred)
- ❌ Audit logging (deferred)

## 10. API Specification

### POST /auth/login
```json
// Request
{ "username": "alice", "password": "pass123" }

// Response 200
{ "access_token": "<jwt>", "token_type": "bearer", "role": "finance" }

// Response 401
{ "detail": "Incorrect username or password" }
```

### GET /auth/me
```
Authorization: Bearer <token>
```
```json
// Response 200
{ "username": "alice", "role": "finance" }
```

### POST /chat/query
```
Authorization: Bearer <token>
```
```json
// Request
{ "query": "What was our Q3 gross margin?" }

// Response 200
{
  "answer": "The Q3 2024 gross margin was 62%...",
  "sources": [
    { "file": "quarterly_financial_report.md", "section": "Q3 2024 Financial Summary" }
  ],
  "role": "finance"
}

// Response 403 (role has no access to queried data)
{ "detail": "Access denied" }
```

### GET /health
```json
{ "status": "ok" }
```

## 13. Future Considerations

- **Persistent chat history** — store conversation turns in PostgreSQL per user session
- **Admin UI** — manage users, roles, and document uploads without code changes
- **Document upload** — drag-and-drop ingestion with automatic department tagging
- **OAuth2 / SSO** — integrate with Google Workspace or Azure AD for enterprise auth
- **Field-level access control** — hide specific columns (e.g., salary) from subset of HR users
- **Streaming responses** — SSE or WebSocket for real-time token streaming
- **Re-ranking** — add a cross-encoder re-ranker between retrieval and generation for better precision
- **Multi-language support** — localized queries and responses
- **Audit logging** — record who queried what and when for compliance

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Pinecone metadata filter bypass | Low | High | Enforce filter server-side only; never trust client-supplied filters |
| LLM hallucination beyond retrieved context | Medium | Medium | Prompt instructs model to answer from context only; Ragas faithfulness score monitors this |
| Google embedding API rate limits during ingestion | Low | Medium | Batch embedding calls; add retry logic with exponential backoff |
| Groq API downtime | Low | High | Add error handling with user-friendly fallback message; consider model fallback |
| JWT secret exposure | Low | Critical | Rotate secret immediately if exposed; use strong random value in production |
